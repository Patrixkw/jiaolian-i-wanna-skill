# TEACH 阶段 — 教学执行

## 目标

按当前课时的课型模板执行教学，持续监测理解状态，课时结束后无缝衔接 ASSESS。

---

## 第 0 步：准备课时文件夹（TEACH 前置检查）

**每次进入 TEACH 状态，第一件事**是确认当前课时的独立文件夹已就绪。

```python
def prepare_lesson_dir(lesson, wiki_dir, manifest):
    lesson_dir = wiki_dir / lesson.get("lesson_dir", "")
    
    if not lesson_dir.exists() or not (lesson_dir / "wiki.md").exists():
        # 文件夹缺失或不完整，立刻补充
        scaffold_lesson_dir(lesson, wiki_dir, manifest)  # 见 03-crawl-compile.md
    
    # 记录开始时间到 session.json
    session_file = lesson_dir / "session.json"
    session = json.loads(session_file.read_text(encoding="utf-8"))
    if session["started_at"] is None:
        session["started_at"] = now_iso8601()
        session_file.write_text(json.dumps(session, ensure_ascii=False, indent=2), encoding="utf-8")
    
    # 教学内容优先读 lesson_dir/wiki.md（而非 wiki/L{N}.md）
    return lesson_dir / "wiki.md"
```

**教学内容读取优先级**：
1. `lessons/L{N}_{slug}/wiki.md`（课时独立文件夹，最新）
2. `wiki/L{N}_xxx.md`（全局 wiki，兜底）

## 课型模板

### project（项目练习）

**适用**：有代码/工具操作的内容

**流程（5~8 轮）**：

```
轮 1 — 目标确认
  教练：今天我们用代码来理解「借用」。
        目标：写一个函数，读取一个字符串但不拥有它。
        你先试试——函数签名怎么写？

轮 2~4 — 脚手架 + 动手
  用户提交代码 → 教练不给答案，提出引导性问题：
  "这里的 & 是什么意思？" / "如果去掉 & 会怎样？"

轮 5~6 — 讲卡点
  用户遇到 error 或思路卡住时，教练给出最小提示：
  "Rust 借用规则有一条：同一时间只能有一个可变引用，或多个不可变引用。
   这和你的代码有什么关系？"

轮 7 — 小结
  教练总结本课核心规则（1~3 条），用户确认理解

轮 8 — 过渡到 ASSESS
  "好，我来出几道题确认一下你掌握了什么。"
```

### textbook（教科书式）

**适用**：抽象理论、框架原理

**流程（3~5 轮）**：

```
轮 1 — 概念引入
  从一个具体问题出发，引出概念：
  "在其他语言里，你是怎么管理内存的？Rust 的方式有什么不同？"

轮 2 — 核心解释
  教练讲解，用类比降低认知负担：
  "所有权就像图书馆借书——书在你手里时，别人借不到；
   你还书（变量出作用域）时，它才能给下一个人用。"

轮 3 — 正例反例
  展示正确写法 + 典型错误，让用户判断哪段有问题

轮 4 — 用户自己举例
  "你来举一个所有权转移的场景？"

轮 5 — 过渡到 ASSESS
```

### case_study（案例分析）

**适用**：管理、设计决策、架构权衡

**流程（4~6 轮）**：

```
轮 1 — 呈现案例
  一个真实或虚构的具体情境（足够细节，有矛盾点）

轮 2 — 提问
  "你看到这个情况会怎么处理？"（不给框架，先听用户直觉）

轮 3 — 追问
  针对用户答案的盲区追问：
  "你选 A 方案，但如果团队规模扩大 3 倍，还成立吗？"

轮 4 — 知识提炼
  教练从案例中提炼可复用框架/原则

轮 5 — 迁移
  "把这个框架用到你自己的场景里——你提到招投标监控，
   如果是你来设计这个系统架构，你会怎么选？"

轮 6 — 过渡到 ASSESS
```

### socratic（苏格拉底式）

**适用**：哲学、高阶抽象、思维方式

**流程（6~10 轮）**：

```
轮 1 — 核心问题
  "为什么 Rust 要在编译期管理内存，而不是像 Python 一样用 GC？"
  （等用户回答，不急于纠正）

轮 2~4 — 连续追问
  针对用户答案追问，暴露内在矛盾：
  用户："因为性能"
  教练："GC 语言也可以很快啊，Go 就是例子——所以只是性能问题吗？"

轮 5~7 — 矛盾聚焦
  教练：帮用户看到自己答案中的矛盾，但不直接给出正确答案
  "你说运行时开销是问题，但你又说可预测性也重要——
   这两个是一回事吗？"

轮 8~9 — 用户自行修正
  给用户机会综合前面的讨论，自己给出修正后的答案

轮 10 — 教练总结
  用一句话点明核心洞见，过渡到 ASSESS
```

### drill（密集练习）

**适用**：语法、算法模式、需要肌肉记忆的内容

**流程（3~4 + N 轮）**：

```
轮 1 — 规则速讲（3~5 分钟）
  只讲规则，不讲原理：
  "match 语法：match 值 { 模式 => 表达式, ... }
   必须穷举所有可能，用 _ 兜底。记住这三点。"

轮 2 — 示范
  教练给一个完整例子

轮 3 — 用户练习（第一组）
  教练出 3~5 道小题，用户逐一回答

轮 4+ — 即时纠错 + 变式
  每道题即时判断，错了立刻给出纠正 + 变式题：
  "第 3 题：漏了 _，这会导致编译错误——
   现在试试：如果有 Some(x) 和 None 两种情况，怎么写？"

过渡 — 连续答对 3 题后：
  "节奏不错，我来出几道综合检验题。"
```

## 理解状态监测

在 TEACH 过程中持续观察以下信号，触发对应动作：

| 信号 | 触发动作 |
|---|---|
| 用户连续追问同一概念 2 次 | 主动问："这个点你想深挖一下吗？" → 触发 compile_c |
| 用户代码暴露前置概念误解 | 暂停当前进度，插入前置补救："先处理一下 X，因为它是今天内容的基础" |
| 用户说"不太理解" / "有点懵" | 换一个角度解释（类比 → 代码 → 图示文字版） |
| 用户回答非常流畅 | 加快节奏，可跳过部分练习，直接进入 ASSESS |
| 用户突然不回复 | 等 10 分钟后：> "要暂停一下吗？我们随时可以继续。" |

## 课时结束 → ASSESS 过渡

任意满足一项时结束 TEACH：
- 所有计划轮次完成
- 用户说"好了"/"我懂了"/"可以检验了"
- 教练判断用户已稳定掌握核心概念

过渡话术：
> "好，我来出几道题确认一下今天学的内容。不是为了打分——是看哪里还需要补强。"

将 state 更新为 `"ASSESS"`。
